## 1. 目标与约束

### 1.1 目标

- 在不破坏“本地优先”的前提下，为系统提供**可选**的实时/增量信息更新能力
- 建立可解释的证据链：知道“信息来自哪里、何时更新、可信度如何”
- 避免将未经验证的外部信息污染长期记忆（KG/semantic memory）

### 1.2 关键约束（强制）

1. **用户授权**：任何联网/抓取行为必须默认关闭，且需用户显式开启与可随时关闭。
2. **最小化上传**：对外请求只发送必要的查询关键词；禁止上传对话全文与个人敏感信息。
3. **可审计**：抓取/解析/写入行为必须可追踪（至少能还原来源 URL/时间/摘要/权重）。
4. **安全边界**：外部内容必须经过安全过滤与事实核查（见 `fact_checker.md` 与 `security_architecture.md`）。

## 2. 组件划分（契约）

建议将实时知识接入拆为 6 个可替换组件：

1. **Source Registry（信源注册表）**：维护来源列表、权重、更新频率、健康状态
2. **Ingestion（摄取）**：RSS/站点/API/文件等来源获取原始文本
3. **Normalization（标准化）**：去噪、编码统一、结构化字段（时间、标题、作者、链接）
4. **Fact Checking（核查）**：多源交叉验证、冲突检测、置信度评估
5. **Summarization（摘要/切块）**：生成可索引的 chunk 与摘要
6. **Storage & Indexing（落盘与索引）**：写入向量库与知识图谱，并维护版本/证据链

## 3. 数据流（端到端）

标准数据流（从外部来源到可检索知识）：

1. Source Registry 选择待更新来源
2. Ingestion 抓取原始内容（RSS/网页/API）
3. Normalization 生成统一结构（示例字段：`source_id/url/published_at/title/body_hash`）
4. Fact Checking：
   - 形成 claim 列表（可用规则 + LLM）
   - 检索对照来源（同主题多源）
   - 输出置信度与冲突标签
5. Summarization：
   - 生成摘要（用于 UI/回溯）
   - 切块（用于向量索引）
6. Storage & Indexing：
   - 向量库写入 chunk embedding（episodic/semantic 视策略）
   - 图谱写入实体/关系（仅对高置信度内容；冲突内容可进入“待核查区”）
7. 审计记录（如启用审计）

## 4. 缓存、去重与版本控制

为避免重复抓取与污染，必须具备：

- **去重键**：`(source_id, url)` 或 `body_hash`
- **增量策略**：仅处理新内容（按发布时间/etag/last_modified）
- **版本控制**：对同一事实/实体允许多版本并存，直到冲突消解或用户确认

## 5. 写入策略（防止“外部信息污染长期记忆”）

建议分三档写入策略：

1. **不写入长期记忆**：仅作为“临时证据”参与当前回答（低置信度/高风险内容）
2. **写入 episodic（可遗忘）**：写入向量库但设置较低 importance 与更快衰减
3. **写入 semantic/KG（长期）**：仅对高置信度 + 多源一致的内容写入 KG，并记录证据链

## 6. 安全与隐私

- **敏感信息过滤**：抓取内容进入系统前先做内容安全扫描（政治/暴力/色情/PII 等）
- **网络访问策略**：
  - 推荐通过受控网络层（代理/限流/域名白名单）
  - 默认阻止插件或模型直接发起任意网络请求
- **用户数据权利**：用户可导出/删除“实时知识缓存与索引”，并可一键关闭联网功能

## 7. 验收清单（建议）

最小可验收用例（端到端）：

1. 开启实时知识开关 → 成功抓取 RSS → 生成摘要 → 写入向量库 → 可检索命中
2. 多源冲突内容 → 标记冲突 → 不写入 KG（或写入待核查区）→ 回答中提示“不确定”
3. 关闭实时知识开关 → 任何联网请求被禁止/短路 → 系统退回本地知识与记忆
