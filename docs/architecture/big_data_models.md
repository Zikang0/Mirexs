# 大数据模型技术设计

## 1. 引言

本文档详细阐述 Mirexs 项目中大数据模型（特别是大型语言模型 LLM）的技术设计，涵盖模型选型、量化策略、微调方法、并行推理以及模型管理与更新机制。旨在为 Mirexs 的认知核心层提供强大的语言理解与生成能力，并确保其在不同硬件环境下的高效运行。

## 2. 模型选型策略

Mirexs 采用多模型路由策略，根据任务需求和硬件资源动态选择最合适的 LLM。核心选型原则包括：

*   **性能与效率平衡**：兼顾模型在特定任务上的表现和推理速度。
*   **硬件兼容性**：支持从消费级 GPU 到专业级加速卡的不同硬件配置。
*   **开源生态**：优先选择拥有活跃社区支持和丰富工具链的开源模型。
*   **多语言能力**：支持中文及其他主流语言。

### 2.1 主力模型列表

| 模型名称 | 主要用途 | 核心优势 | 推荐部署方式 |
|:---|:---|:---|:---|
| **Llama 3.1 70B** | 日常聊天、情感陪伴、通用推理 | 强大的通用能力，情感理解力强，社区支持广泛 | 量化后部署，如 GGUF/EXL2 |
| **Qwen 3.5 32B** | 日常聊天、情感陪伴、通用推理 | 针对中文优化，性能优异，长上下文能力 | 量化后部署，如 GGUF/EXL2 |
| **DeepSeek V3** | 编程、复杂推理、逻辑分析 | 卓越的编程能力和数学推理能力 | 量化后部署，如 GGUF/EXL2 |
| **Qwen3-Coder-32B** | 编程、复杂推理、逻辑分析 | 针对编程任务优化，代码生成与理解能力强 | 量化后部署，如 GGUF/EXL2 |
| **Llama 4 Maverick / Qwen3-Next** | 长上下文 (>128K)、复杂文档分析 | 极长的上下文窗口，适合处理大量信息 | 需高性能硬件，或采用分段处理 |
| **Llama 4 Scout / Qwen3-Omni** | 多模态（图文/视频）理解与生成 | 强大的多模态融合能力，理解复杂视觉信息 | 需高性能硬件，或采用多模态模型服务 |

## 3. 模型量化策略

为确保 Mirexs 在不同硬件（尤其是消费级 GPU）上的高效运行，模型量化是关键技术。Mirexs 主要采用以下量化技术：

### 3.1 GGUF (GPT-Generated Unified Format)

*   **描述**：由 `llama.cpp` 项目推广的统一模型格式，支持多种量化级别（如 Q4_K_M, Q5_K_M, Q8_0）。
*   **优势**：广泛的硬件兼容性，尤其适合 CPU 推理和低显存 GPU。易于部署和使用。
*   **应用场景**：作为 Mirexs 在资源受限环境（如笔记本电脑）下的首选量化格式，确保基本功能可用。

### 3.2 EXL2

*   **描述**：一种高效的 2-bit 量化格式，旨在提供极高的压缩率同时保持较好的模型性能。
*   **优势**：在极低显存占用下提供接近 FP16 的性能，推理速度快。
*   **应用场景**：在显存紧张但对性能有较高要求的场景下，如中低端 GPU。

### 3.3 量化流程

1.  **模型下载**：从 Hugging Face 等平台下载原始 FP16/BF16 模型。
2.  **量化转换**：使用 `llama.cpp` 工具链或 `exllamav2` 库进行量化转换。
3.  **兼容性测试**：对量化后的模型进行性能和准确性测试，确保满足要求。
4.  **智能按需下载**：`infrastructure/model_manager.py` 根据硬件检测结果，自动下载最适合的量化模型版本。

## 4. 模型微调方法

Mirexs 支持通过微调来提升模型在特定任务上的表现和个性化能力。

### 4.1 LoRA (Low-Rank Adaptation)

*   **描述**：一种参数高效的微调技术，通过在预训练模型中注入小的可训练矩阵来适应新任务，而无需更新所有模型参数。
*   **优势**：训练成本低，显存占用少，易于部署和切换。
*   **应用场景**：个性化定制（用户偏好、特定领域知识）、新技能学习、少量数据下的任务适应。

### 4.2 QLoRA (Quantized LoRA)

*   **描述**：在 LoRA 的基础上，进一步对预训练模型进行量化，从而在微调过程中显著降低显存消耗。
*   **优势**：在消费级 GPU 上进行大型模型微调成为可能，进一步降低硬件门槛。
*   **应用场景**：与 LoRA 类似，但适用于显存更受限的环境。

### 4.3 微调数据与流程

1.  **数据收集**：从用户交互、特定领域语料中收集高质量的指令-响应对。
2.  **数据清洗与标注**：确保数据质量，必要时进行人工标注。
3.  **微调训练**：使用 `peft`、`transformers` 等库进行 LoRA/QLoRA 训练。
4.  **模型合并与部署**：将 LoRA 适配器与基座模型合并，或作为独立适配器加载，并通过 `model_manager.py` 进行管理。

## 5. 模型并行推理

对于大型模型或高并发场景，Mirexs 采用并行推理技术以提升吞吐量和降低延迟。

### 5.1 张量并行 (Tensor Parallelism)

*   **描述**：将模型的每一层（如线性层、注意力层）的权重和计算分散到多个 GPU 上，每个 GPU 只处理部分张量。
*   **优势**：有效处理单 GPU 无法加载的超大模型。
*   **应用场景**：Llama 3.1 70B 等大型模型在多卡服务器上的部署。

### 5.2 流水线并行 (Pipeline Parallelism)

*   **描述**：将模型的不同层分配给不同的 GPU，形成一个处理流水线。每个 GPU 顺序处理其分配到的层。
*   **优势**：进一步减少显存占用，提高整体吞吐量。
*   **应用场景**：与张量并行结合，处理更大规模的模型或更高并发请求。

### 5.3 推理引擎

*   **vLLM**：高性能 LLM 推理库，支持 PagedAttention 等优化技术，显著提升吞吐量。
*   **llama.cpp**：轻量级推理引擎，支持 GGUF 格式，在 CPU 和低端 GPU 上表现优异。
*   **transformers**：Hugging Face 官方库，提供灵活的模型加载和推理接口。

## 6. 模型管理与更新机制

Mirexs 建立了一套完善的模型管理与更新机制，确保模型的生命周期管理和持续优化。

### 6.1 模型库 (Model Hub)

*   **中心化存储**：所有模型权重、量化版本、微调适配器统一存储在 `data/models/` 目录下，并进行版本控制。
*   **元数据管理**：`infrastructure/model_hub/model_configs.yaml` 记录模型的 ID、标签、VRAM 需求、上下文窗口、量化格式等元数据。

### 6.2 智能按需下载与加载

*   `infrastructure/model_manager.py` 负责根据 `hardware_detector.py` 检测到的硬件能力和当前任务需求，自动选择并下载最合适的模型版本。
*   支持模型热加载和卸载，优化显存使用。

### 6.3 持续集成与部署 (CI/CD)

*   **模型训练流水线**：自动化模型训练、评估、量化和打包。
*   **模型部署流水线**：自动化模型版本发布、A/B 测试和回滚机制。

## 7. 性能指标与优化

*   **推理延迟**：日常聊天 P95 ≤ 800ms，复杂推理 P95 ≤ 4.5s。
*   **吞吐量**：通过批处理、KV Cache 优化、模型并行等技术提升。
*   **显存占用**：通过量化、模型卸载、高效推理引擎等技术降低。
