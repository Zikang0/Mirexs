# Mirexs 文档状态规范

## 1. 目的

本规范用于统一 `docs/` 下所有 Markdown 文档的状态声明、更新时间、代码映射关系和维护责任，避免出现“文档写得很完整，但代码并不存在”或“代码已调整，文档仍停留在旧版本”的情况。

后续所有开发文档都应满足两个目标：

1. 可读。新成员能够通过文档快速理解模块边界、输入输出和依赖。
2. 可执行。研发、测试、评审都能直接依据文档判断实现是否符合预期。

## 2. 适用范围

本规范适用于 `docs/**/*.md` 的正式文档，包含但不限于：

- `docs/architecture/` 架构设计文档
- `docs/developer_guide/` 开发指南
- `docs/technical_specifications/` 技术规范
- `docs/api_reference/` 接口参考
- `docs/user_manual/` 用户手册

以下文件可以不完全遵守本规范，但仍建议补充状态说明：

- 会议纪要
- 历史归档文档
- 自动生成的报告
- 市场/法务草案

## 3. 推荐 Front Matter

每份正式文档建议在头部使用如下 YAML front matter：

```yaml
---
status: planned | partial | implemented | deprecated | archived
last_reviewed: 2026-03-30
corresponds_to_code: "真实相对路径、多个路径用逗号分隔，或写 暂无"
related_issues: ""
references: ""
---
```

字段说明：

- `status`：文档与代码的当前对齐状态
- `last_reviewed`：最后一次人工校对日期
- `corresponds_to_code`：该文档实际对应的代码、配置或脚本入口
- `related_issues`：相关任务、缺陷或待办编号
- `references`：依赖阅读的其他文档

## 4. 状态定义

### 4.1 `planned`

表示该文档描述的是目标设计，代码尚未开始或仓库中只有目录级占位。

使用要求：

- 必须明确写出“规划态”
- 必须说明落地前置条件
- 不得使用“已支持”“生产可用”之类结论性表述

### 4.2 `partial`

表示代码已存在，但实现与文档仍有差异，或仅完成部分功能。

使用要求：

- 必须标出已完成内容与未完成内容
- 必须指出当前生效口径以代码/配置为准的文件路径
- 必须给出后续补齐方向

### 4.3 `implemented`

表示文档与当前代码、配置和行为已经基本一致，可作为开发和验收依据。

使用要求：

- 关键路径必须有代码映射
- 输入输出、错误处理、配置项需写清楚
- 变更时必须同步更新文档

### 4.4 `deprecated`

表示该方案或接口仍可能存在，但不建议继续使用。

使用要求：

- 必须写出替代方案
- 必须说明预计下线时间或触发条件

### 4.5 `archived`

表示仅用于保留历史记录，不再作为当前研发依据。

使用要求：

- 必须在开头注明“归档文档”
- 不得再被其他正式文档作为唯一权威引用

## 5. 每类文档至少应包含的内容

### 5.1 架构文档

至少包含：

- 模块职责
- 分层边界
- 输入输出数据结构
- 当前实现入口
- 未完成项与风险
- 验收标准

### 5.2 开发指南

至少包含：

- 适用角色
- 开发前准备
- 目录与代码入口
- 操作步骤
- 常见错误
- 完成定义

### 5.3 技术规范

至少包含：

- 术语定义
- 强制约束
- 兼容策略
- 配置口径
- 测试/验收规则

### 5.4 用户手册

至少包含：

- 适用对象
- 使用前提
- 操作步骤
- 结果说明
- 常见问题

## 6. 文档编写规则

- 不写不存在的模块、脚本、端点和配置项
- 如果是目标态，必须明确写“规划”“建议”“目标”
- 所有路径使用仓库真实相对路径
- 所有性能指标必须说明测试环境或仅为目标值
- 所有 API 文档必须与 `docs/technical_specifications/api_specification.md` 和 `docs/technical_specifications/api_envelope_standard.md` 保持一致

## 7. 文档更新触发条件

出现以下任一情况，必须同步更新文档：

- 新增模块、目录或配置文件
- 修改接口路径、字段名或返回结构
- 变更核心依赖和运行环境要求
- 调整默认模型、路由策略或安全策略
- 修复会影响用户使用方式或部署方式的问题

## 8. 审核清单

文档提交前至少检查以下问题：

- 文档状态是否与代码真实状态一致
- 文中引用的路径是否真实存在
- 是否说明了当前限制、风险和待补齐内容
- 是否能让不了解上下文的开发者直接开始实现
- 是否与 README、requirements 和其他子文档口径一致

## 9. 维护原则

文档不是宣传材料，而是开发契约。凡是会影响实现、测试、联调、部署和运维的内容，都必须优先保证准确，而不是“看起来完整”。
