# 代码评审流程

## 1. 评审目标

代码评审的目标不是挑风格问题，而是尽早发现行为错误、边界破坏、回归风险、安全缺陷和不可维护设计。

## 2. 进入评审前的提交要求

提交人至少需要提供：

- 变更背景
- 改动范围
- 验证方式
- 已知限制
- 是否修改接口、配置、数据结构或默认行为

如果是 UI 或交互变更，还应提供截图、录屏或用户路径说明。

## 3. 评审流程

1. 提交人自检：确认代码、文档、配置和测试已同步。
2. 自动检查：运行最小必要测试、静态检查和安全扫描。
3. 人工评审：按正确性、边界、性能、安全和可维护性审查。
4. 反馈回合：提交人根据意见修改并更新验证结果。
5. 准入判断：无阻塞问题后再合并。

## 4. 评审重点

### 4.1 正确性

- 逻辑是否满足需求
- 边界条件是否处理
- 异常路径是否可控

### 4.2 架构边界

- 是否违反分层依赖
- 是否把临时逻辑写进了公共模块
- 是否出现跨模块契约漂移

### 4.3 性能与资源

- 是否引入阻塞 I/O
- 是否引入不必要的大对象、重复加载或高频扫描
- 是否会导致 GPU、内存或存储资源异常增长

### 4.4 安全与隐私

- 是否暴露敏感信息
- 是否绕过鉴权或输入校验
- 是否缺少审计和风险收敛

### 4.5 文档一致性

- 文中提到的路径、接口和配置是否真实存在
- 行为变更是否同步到架构、接口和技术规范

## 5. 问题严重度

- `P0`：会导致严重安全事故、数据损坏或主链路不可用
- `P1`：功能错误或明显回归
- `P2`：性能、稳定性、可维护性问题
- `P3`：风格、命名或轻微文档问题

## 6. 评审结论类型

- `approve`：可以合并
- `request changes`：存在阻塞问题，需要修改
- `comment`：非阻塞建议，可由提交人判断是否采纳

## 7. 评审人检查清单

- 改动是否聚焦
- 是否补充或更新测试
- 是否有明确回退策略
- 是否更新了相关文档
- 是否引入新的环境要求或外部依赖

## 8. 提交人响应要求

- 对每条阻塞意见都要明确回应
- 如不采纳建议，必须说明理由
- 修改后应重新给出验证结果，而不是只说“已修复”
