# 测试指南

## 1. 目标

本文件定义 Mirexs 项目的测试层级、最小覆盖要求和开发者提测原则。测试的目标不是追求形式化覆盖率，而是用最小成本尽早发现功能错误和回归风险。

## 2. 测试层级

- 单元测试：验证单个类、函数或模块的纯逻辑
- 集成测试：验证模块之间的接口契约和协同工作
- 性能测试：验证关键链路的延迟、吞吐和资源消耗
- 安全测试：验证输入校验、鉴权、权限边界和敏感路径

## 3. 目录约定

- `tests/unit_tests/`
- `tests/integration_tests/`
- `tests/performance_tests/`
- 如新增新的测试子目录，需补充说明其目标和执行方式

## 4. 开发者提测原则

- 新增功能至少提供单元测试或最小可执行验证
- 修复缺陷时，优先补回归测试
- 改接口、改配置、改默认行为时，要考虑是否需要集成测试
- 改模型路由、缓存、资源管理时，要考虑性能验证

## 5. 运行方式

```bash
pytest tests/
```

如果项目环境复杂，建议按子目录或标签运行最相关测试，再逐步扩大范围。

## 6. 覆盖重点

以下模块应优先保证测试质量：

- API 入口和请求校验
- 模型路由与资源管理
- 知识图谱与记忆检索
- 安全过滤与审计逻辑
- 关键配置解析和回退逻辑

## 7. 测试结果要求

提交 PR 前至少说明：

- 运行了哪些测试
- 哪些测试因环境限制未运行
- 是否使用了替代验证方式
- 是否存在残余风险

## 8. 性能验证要求

对性能敏感改动，至少需要说明：

- 测试环境
- 测试场景
- 关键指标
- 与目标值或旧版本的差异

## 9. 不合格的测试提交

- 只改代码不验证
- 用截图替代测试结论
- 把手工验证写成“已测试通过”，但没有步骤
- 明显破坏现有契约却没有补充相应测试
