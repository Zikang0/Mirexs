# 开发者贡献指南

## 1. 目标

本文件说明开发者如何在 Mirexs 仓库中选择任务、开展开发、提交变更和完成交付。

## 2. 适合贡献的主题

- 基础设施层空壳补齐
- 配置和接口契约完善
- 模型接入、路由和资源管理
- 安全、测试和部署能力增强
- 文档与代码一致性修复

## 3. 开发前准备

开始编码前，至少要完成：

- 阅读对应模块的架构文档
- 阅读相关开发指南或技术规范
- 确认当前仓库中真实存在的代码入口
- 明确该任务是否需要同步更新文档和测试

## 4. 分支与提交建议

- `main` 视为稳定主线
- 建议使用 `feature/...`、`fix/...`、`docs/...`、`refactor/...`
- Commit 建议使用 Conventional Commits，例如 `feat: add hardware-aware model routing`

## 5. 标准工作流程

1. 从 Issue 或明确需求开始。
2. 阅读相关代码和文档，确认现状。
3. 以最小范围实现变更。
4. 更新受影响文档。
5. 完成最小验证。
6. 提交 PR 并说明影响范围。

## 6. 文档同步要求

如果改动涉及以下任一项，必须同步更新文档：

- 目录结构
- API
- 配置字段
- 默认模型或资源要求
- 安全策略
- 部署方式

## 7. 提交 PR 时建议附带的信息

- 改动背景
- 方案说明
- 关键文件路径
- 验证步骤
- 已知限制

## 8. 不建议的贡献方式

- 直接在不了解架构边界的情况下大规模重构
- 把文档写成目标态，但不说明代码尚未实现
- 提交无法运行、无法验证、无法说明的半成品
