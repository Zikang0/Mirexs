# 编码规范

## 1. 适用范围

本规范适用于仓库中的 Python 代码、配置加载逻辑、文档示例代码和对外接口实现。

## 2. 基础风格

- 遵循 PEP 8，但优先保证可读性
- 模块、类、函数必须有明确职责
- 公共函数和跨模块对象必须有类型注解
- 模块级 docstring 必须说明该文件负责什么、不负责什么

## 3. 命名规则

- 类名使用 `PascalCase`
- 函数、变量、模块使用 `snake_case`
- 常量使用 `UPPER_CASE`
- 枚举成员命名应稳定且可序列化，不要频繁变化

## 4. 结构化要求

- 公共数据结构优先使用 `dataclass` 或 Pydantic
- 配置读取集中处理，不要在业务逻辑中分散读取文件
- 避免把初始化、业务逻辑和 CLI/脚本入口写在同一个模块中

## 5. 错误处理

- 所有外部 I/O 必须显式捕获和转换异常
- 不允许无说明地吞掉异常
- 对外暴露的错误信息应稳定、可解释且不泄露敏感细节
- 业务异常和基础设施异常应分层定义

## 6. 日志要求

- 统一使用 `logging`
- 日志要记录上下文，不要只打印“失败了”
- 生产日志不得输出密钥、口令、令牌、原始敏感数据
- 对高频路径注意日志量，避免把日志写成新的性能瓶颈

## 7. 异步与性能要求

- 网络、数据库、磁盘等 I/O 优先使用异步或线程隔离方案
- 高成本初始化应支持缓存、延迟加载或复用
- 模型、数据库连接和外部资源要有生命周期管理
- 对高频调用路径尽量避免重复扫描文件系统和重复解析大配置

## 8. 文档与注释要求

- 对外函数、复杂函数和关键类应有 docstring
- 注释只解释设计理由、边界条件和易错点
- 新增功能、路径、配置或接口时必须同步更新 `docs/`

## 9. 禁止事项

- 硬编码账号、密码、绝对个人路径
- 把演示代码直接当成生产代码提交
- 使用不可追溯的魔法常量
- 通过复制粘贴形成多个几乎一致的实现而不抽象

## 10. 完成定义

代码合入前至少满足：

- 结构清晰
- 类型和异常可说明
- 路径与文档一致
- 有最小验证
- 没有明显的占位逻辑冒充正式实现
