# Mirexs Issue 提交指南

## 1. 目标

本文件定义 Issue 的信息收集标准，确保维护者能快速判断问题类别、影响范围和复现条件。一个合格的 Issue 应该能让开发者不依赖额外上下文就开始排查。

## 2. Issue 分类

建议至少使用以下几类：

- Bug：已有能力行为错误或回归
- Feature Request：新增能力或增强需求
- Documentation：文档错误、缺失或与代码不一致
- Performance：延迟、资源占用、吞吐异常
- Security：越权、泄露、输入绕过、审计缺失

## 3. Bug 报告必须包含的信息

- 期望行为与实际行为
- 最小复现步骤
- 出现频率：必现、偶发或特定环境出现
- 环境信息：
  - 操作系统与版本
  - Python 版本
  - GPU 型号与显存
  - 关键依赖版本
  - 相关配置项
- 关键日志、报错栈、截图或终端输出
- 如果涉及接口，附上请求参数和返回结果

## 4. 功能需求建议模板

- 需求背景：为什么要做
- 使用场景：谁在什么条件下使用
- 约束条件：性能、隐私、离线、成本、安全
- 预期接口或行为
- 验收标准
- 影响模块与文档范围

## 5. 文档问题建议模板

- 文档路径
- 具体段落或标题
- 错误类型：路径错误、描述过时、实现不一致、约束不完整
- 对应代码或配置证据
- 期望修正方向

## 6. 高质量 Issue 的判断标准

- 问题描述聚焦，不混入多个无关主题
- 提供了足够的上下文，不需要维护者反复追问基础信息
- 区分“当前行为有 bug”与“希望支持新能力”
- 明确是否影响接口、数据、模型、部署或安全

## 7. 常见低质量问题

- 只有“不能用了”“有 bug”而没有复现信息
- 没写环境信息
- 不提供错误日志或关键配置
- 把多个独立问题堆在同一个 Issue 中
- 用 README 宣传语替代实际问题描述

## 8. 建议关联内容

如果已知相关内容，建议在 Issue 中一并关联：

- 对应代码路径
- 对应文档路径
- 相关提交、PR 或历史问题
- 临时绕过方案
