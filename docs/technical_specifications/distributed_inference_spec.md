# 分布式推理规范

## 1. 引言

随着 Mirexs 认知核心层对大型语言模型（LLM）和多模态模型依赖的加深，单设备推理的性能瓶颈日益凸显。为了满足高并发、低延迟的推理需求，并充分利用多卡、多节点集群资源，本文档详细阐述 Mirexs 分布式推理的技术规范，涵盖负载均衡、模型分片、并行计算、容错机制以及性能优化策略。本规范旨在指导 Mirexs 在生产环境中实现高效、稳定的模型推理服务。

## 2. 设计目标与原则

### 2.1 设计目标

*   **高吞吐量**：在单位时间内处理更多的推理请求。
*   **低延迟**：最小化单个推理请求的响应时间。
*   **高可用性**：确保推理服务在部分节点故障时仍能持续运行。
*   **可扩展性**：能够根据业务需求弹性伸缩推理资源。
*   **资源高效利用**：充分利用集群中的 GPU、CPU 和内存资源。
*   **异构硬件支持**：兼容不同型号和性能的加速卡。

### 2.2 设计原则

*   **透明性**：对上层应用屏蔽分布式推理的复杂性。
*   **弹性伸缩**：支持推理服务的自动扩缩容。
*   **容错性**：具备故障检测和自动恢复机制。
*   **模块化**：各组件职责清晰，易于维护和升级。
*   **性能优先**：在保证稳定性的前提下，最大化推理性能。

## 3. 分布式推理架构

Mirexs 分布式推理架构基于微服务理念，将推理服务解耦为多个独立组件，并通过智能路由和负载均衡实现协同工作。

```mermaid
graph TD
    A[Mirexs 应用接口层] --> B(API 网关/负载均衡器);;
    B --> C{推理服务集群};;
    C --> D[模型管理服务];;
    C --> E[推理节点 (GPU/CPU)];;
    E --> F[模型缓存/卸载];;
    E --> G[模型并行/流水线并行];;
    D --> E;

    subgraph 推理服务集群
        C1[推理服务实例 1] --> E;
        C2[推理服务实例 2] --> E;
        C3[推理服务实例 N] --> E;
    end

    subgraph 推理节点
        E1[GPU 1] --> G;
        E2[GPU 2] --> G;
        E3[CPU];;
    end
```

## 4. 核心组件与技术

### 4.1 负载均衡与请求路由

*   **职责**：将推理请求分发到可用的推理服务实例，并根据任务类型、模型需求和节点负载进行智能路由。
*   **技术选型**：
    *   **API 网关**：如 Nginx, Envoy，提供统一入口，并可集成认证、限流等功能。
    *   **智能路由**：`multi_model_router.py` 模块，根据以下策略进行请求分发：
        *   **模型兼容性**：请求所需的模型是否已加载到目标节点。
        *   **硬件资源**：目标节点的 GPU 显存、CPU 利用率等。
        *   **任务优先级**：高优先级任务优先分配给空闲或高性能节点。
        *   **模型版本**：确保请求路由到正确的模型版本。
    *   **负载均衡算法**：轮询 (Round Robin)、最少连接 (Least Connection)、加权轮询 (Weighted Round Robin) 等。

### 4.2 模型管理服务 (Model Management Service)

*   **职责**：负责模型的生命周期管理，包括模型的加载、卸载、版本控制和状态监控。
*   **核心功能**：
    *   **模型注册与发现**：维护所有可用模型的元数据（如模型 ID、版本、所需资源、加载状态）。
    *   **动态加载/卸载**：根据推理请求和资源情况，动态加载或卸载模型到推理节点。
    *   **模型缓存**：将常用模型或近期使用的模型缓存到显存或内存中，减少加载时间。
    *   **模型版本控制**：支持多版本模型共存和灰度发布。
    *   **健康检查**：定期检查推理节点上模型的运行状态。

### 4.3 推理节点 (Inference Node)

*   **职责**：执行实际的模型推理计算。
*   **硬件配置**：
    *   **GPU 节点**：配备高性能 GPU（如 NVIDIA A100, H100 或消费级 RTX 系列），用于 LLM 和多模态模型的加速推理。
    *   **CPU 节点**：用于轻量级模型、预处理/后处理任务或作为 GPU 节点的备用。
*   **推理引擎**：
    *   **vLLM**：用于 LLM 的高性能推理，支持 PagedAttention、连续批处理等优化技术，显著提升吞吐量和降低延迟。
    *   **TensorRT/OpenVINO**：针对特定硬件平台进行模型优化和加速。
    *   **llama.cpp**：用于 GGUF 格式模型的 CPU/GPU 推理，尤其适用于资源受限环境。

### 4.4 模型并行与流水线并行

*   **职责**：在单个推理请求需要跨多个 GPU 或节点执行时，协调模型的计算。
*   **技术实现**：
    *   **张量并行 (Tensor Parallelism)**：将模型的权重和计算在层内进行切分，分配到不同的 GPU 上。适用于单卡无法容纳的超大模型。
    *   **流水线并行 (Pipeline Parallelism)**：将模型的不同层分配到不同的 GPU 上，形成一个计算流水线。可以与张量并行结合使用，进一步提高资源利用率。
    *   **数据并行 (Data Parallelism)**：在处理批量请求时，将不同的请求分发到不同的推理节点上并行处理。适用于高吞吐量场景。
*   **通信优化**：使用 NCCL (NVIDIA Collective Communications Library) 或 Gloo 等库进行高效的 GPU 间通信。

### 4.5 容错机制

*   **节点健康检查**：定期对推理节点进行健康检查，包括硬件状态、服务进程状态、模型加载状态等。
*   **故障转移 (Failover)**：当检测到推理节点故障时，负载均衡器将自动停止向该节点分发请求，并将未完成的请求重新路由到健康的节点。
*   **自动恢复**：通过 Kubernetes 等容器编排工具，实现故障节点的自动重启和恢复。
*   **请求重试**：客户端在接收到推理失败响应时，可配置重试机制。

## 5. 性能指标与优化

### 5.1 关键性能指标 (KPIs)

*   **吞吐量 (Throughput)**：每秒处理的请求数 (Requests Per Second, RPS)。
*   **延迟 (Latency)**：
    *   **首次令牌时间 (Time To First Token, TTFT)**：从请求发送到第一个输出令牌生成的时间。
    *   **每令牌延迟 (Time Per Token, TPT)**：生成每个后续令牌的平均时间。
    *   **端到端延迟 (End-to-End Latency)**：从请求发送到所有输出令牌生成完成的时间。
*   **资源利用率**：GPU 利用率、显存占用、CPU 利用率。

### 5.2 优化策略

*   **批处理 (Batching)**：将多个推理请求合并为一个批次进行处理，提高 GPU 利用率。
*   **KV Cache 优化**：缓存 LLM 推理过程中生成的 Key 和 Value 向量，避免重复计算。
*   **模型量化**：如 GGUF, EXL2，减少模型大小和显存占用，提高推理速度。
*   **模型剪枝与蒸馏**：减小模型复杂度，提高推理效率。
*   **异步推理**：利用异步 I/O 和多线程/多进程，提高系统并发能力。
*   **硬件加速**：充分利用 Tensor Cores 等专用硬件加速单元。
